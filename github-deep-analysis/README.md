# GitHub 开源项目战略分析师

> 不是技术评估报告，是战略判断报告——每个项目背后都有一群人在押赌注，你的任务是找到赌注、判断对错。

## 解决什么问题

看到一个热门 GitHub 项目，Star 涨得飞快，README 写得漂亮——但你真正想知道的是：

- **这群人到底押了什么赌注？押对还是押错？**
- **这个项目在产业链里占什么位置，谁会为它付费？**
- **和市面上已有的工具比，它到底有什么不一样？**
- **值不值得投入时间/资源去用、去集成、去跟进？**

大多数开源分析停留在技术层面——架构好不好、代码质量高不高、Star 多不多。但一个项目的命运从来不只由技术决定。产业位置、商业可持续性、市场时机、团队背景，甚至伦理风险，才是真正的决定变量。

这个 Skill 产出的是**有判断力的战略分析**，不是面面俱到的技术说明书。

## 核心能力

| 能力 | 说明 |
|------|------|
| **赌注识别** | 不看 README 写了什么，看项目背后押了什么——押对了就是赛道赢家，押错了就是昙花一现 |
| **反常数据挖掘** | Star 高但 commit 少？叙事精致但实现简单？热度增速和 KOL 推文不匹配？反常是洞察的种子 |
| **维度自适应** | 不是每个项目都要分析「技术+市场+商业+人伦」，维度比例由项目的主要矛盾决定 |
| **竞品定位对比** | 市面上已有工具时，自动激活对比框架——白空间识别、替代性分析、可持续差异化判断 |
| **快速可行性评估** | 只需知道「值不值得用/能不能集成」时，走快速路径——兼容性、场景匹配、方案建议，5 分钟出结论 |
| **可验证** | 每个核心判断带证据链和来源链接，读者可以沿着链接自行验证 |

## 使用场景

**场景 1：看到一个项目，想快速判断值不值得关注**
> 给一个 GitHub 链接，几分钟后拿到一份带判断的深度分析——不是「涨知识了」，是「我知道该怎么看这个项目了」。

**场景 2：面对同类工具，不知道选哪个**
> 问「这个和市面上其他工具有什么不一样」，自动激活对比分析——竞品矩阵、空白象限、场景化选择建议。

**场景 3：评估能不能集成到自己的系统**
> 问「值不值得做/能不能集成」，走快速可行性评估——环境兼容性、调用链路、维护成本，直接给结论。

**场景 4：团队需要决策依据**
> 分析结果自动写入飞书文档，团队共享——核心判断前置，论据闭环，读 200 字就知道结论。

## 工作流程

```
GitHub 项目链接
  │
  ├─ 升维检查：用户真实意图是什么？
  │   ├─ 对比辨析 → 竞品定位分析
  │   ├─ 可行性决策 → 快速评估路径
  │   └─ 深度理解 → 全维度战略分析
  │
  ├─ Phase 0：快速扫描，找到主变量
  │   └─ 这个项目的命运最关键地取决于什么？
  │
  ├─ Phase 0.5：发现反常
  │   └─ Star/commit 矛盾、声称vs实际、叙事vs工程、热度vs质量
  │
  ├─ Phase 1：数据采集
  │   └─ README + 元数据 + Issues + Commits + 官网 + KOL 推文
  │
  ├─ Phase 2：形成洞察
  │   └─ 反常 × 主变量 → 核心判断
  │
  └─ 输出：飞书文档（判断前置 + 证据闭环 + 叙事弧线）
```

## 产出样例

每份分析文档遵循一个核心原则：**读者读 200 字就知道结论，后面的内容是让你相信这个结论。**

```
┌─────────────────────────────────────────────────┐
│ 核心判断（含赌注描述）                            │
│ 关键前提：判断在什么条件下成立                     │
│ 失效条件：如果前提不成立，判断怎么变               │
├─────────────────────────────────────────────────┤
│ 一、章节标题（从项目矛盾中提炼，不是维度标签）      │
│   断言 → 证据 → 推论                             │
│   ...                                            │
├─────────────────────────────────────────────────┤
│ 核心判断重述 / 失效条件 / 下一步建议               │
│ Further Reading（3+条，带链接+为什么相关）         │
└─────────────────────────────────────────────────┘
```

## 分析路径说明

| 路径 | 触发信号 | 产出 |
|------|---------|------|
| **全维度战略分析** | 「分析一下」「这个怎么样」 | 完整战略判断报告，章节由项目矛盾驱动 |
| **竞品定位对比** | 「和XX比有什么不一样」「市面上这么多工具」 | 竞品矩阵+白空间+场景选择+差异化可持续性 |
| **快速可行性评估** | 「值不值得做」「能不能集成」 | 兼容性评级+场景匹配+分阶段建议 |

三条路径不是预设选项，是**根据用户意图自动匹配**——发链接时说清楚你想知道什么，系统走对应路径；什么都不说，默认走全维度分析。

## 关键设计决策

| 决策 | 理由 |
|------|------|
| 判断前置，不铺垫 | 读者时间宝贵，200 字内给核心结论，论据在后面 |
| 维度由数据决定，不预设 | 不是每个项目都需要分析技术+市场+商业，信息空洞的维度一笔带过 |
| 章节标题从矛盾中提炼 | 「技术架构分析」是维度标签，不是判断——标题应暗示判断方向 |
| 反常数据优先 | 反常是洞察的种子，Star/commit 矛盾比 README 描述更能揭示真相 |
| 对比信号权重最高 | 用户问「有什么不一样」时，默认走全维度分析是答非所问 |
| 5 个逻辑自检 | 无来源的判断、不可验证的 claim、暗示恶意的词——零容忍 |
