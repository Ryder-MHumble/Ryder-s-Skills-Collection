# 技术项目可行性评估框架

当用户分享一个项目链接并说「这个项目看起来不错」「值不值得做」「能不能参考」时，使用此框架进行系统性评估。

## 核心原则

**不做「看起来不错就集成」的决策。**

评估的核心问题是：用户真正需要的和我实际做的是不是同一件事？

## 评估流程

### Step 1: 项目本质澄清（存活性验证）

先读取项目核心信息，回答：
- 这是什么？（框架/工具/平台/插件？）
- 它依赖什么 runtime？（OpenClaw、Hermes、Python、Node.js、Docker？）
- 它的核心功能是什么？
- 它的部署方式是什么？（一键安装、Docker、代码集成？）

**关键问题：** 用户说的是「参考它的理念」还是「部署它的系统」？这两者的评估路径完全不同。

### Step 2: 与用户环境的兼容性分析

对比项目需求和用户当前环境：

| 维度 | 评估点 |
|------|--------|
| **Runtime** | 项目需要什么 runtime？用户当前用的是什么？是否冲突？ |
| **调用链路** | 项目能否与用户现有 workflow 打通？还是独立系统？ |
| **状态同步** | 项目的实时状态能否被用户感知？还是需要额外维护？ |
| **维护成本** | 部署后需要持续维护什么？（服务、配置、依赖升级） |

**输出：** 可行性评级（⭐⭐⭐⭐⭐ = 即插即用，⭐ = 需要大量桥接工作）

### Step 3: 应用场景与痛点匹配

| 场景 | 项目价值 | 用户当前痛点 | 匹配度 |
|------|---------|-------------|--------|
| 场景 1 | ... | ... | ⭐⭐⭐⭐⭐ |
| 场景 2 | ... | ... | ⭐⭐⭐ |

**关键洞察：** 不要只列项目功能，要对应到用户的实际痛点。如果项目功能很好但解决不了用户的问题，匹配度就是 0。

### Step 4: 可行性分级建议

**方案 A：直接集成（全量采纳）**
- 适用：项目与用户环境完全兼容，且能带来显著价值
- 成本：高（部署、配置、维护）
- 风险：runtime 冲突、长期维护负担

**方案 B：借鉴理念（部分采纳）**
- 适用：项目 governance/设计思想有价值，但系统本身不兼容
- 成本：中（改造现有系统）
- 风险：改造过程中的理解偏差

**方案 C：暂不做（观望/放弃）**
- 适用：项目与用户环境冲突大、维护成本过高、应用场景不明确
- 成本：0
- 风险：错过机会（但可通过持续关注降低）

### Step 5: 分阶段建议

| 阶段 | 动作 | 前提条件 |
|------|------|---------|
| 立即做（0 成本） | 借鉴理念、改造现有配置 | 不需要新系统部署 |
| 评估后做（有成本） | 小规模试点、PoC | 验证核心假设后再投入 |
| 暂不做 | 持续关注、列入观察清单 | 等条件成熟（如 runtime 兼容） |

## 输出格式

```markdown
## 评估结论：[直接集成 / 借鉴理念 / 暂不建议]

### 一、项目本质
[一句话定义 + 核心功能 + 依赖 runtime]

### 二、兼容性分析
[Runtime / 调用链路 / 状态同步 / 维护成本 四维评估]

### 三、应用场景匹配
[场景矩阵：项目价值 vs 用户痛点 vs 匹配度]

### 四、方案对比
| 方案 | 可行性 | 成本 | 风险 |
|------|--------|------|------|
| A. 直接集成 | ⭐⭐ | 高 | runtime 冲突 |
| B. 借鉴理念 | ⭐⭐⭐⭐ | 中 | 改造偏差 |
| C. 暂不做 | - | 0 | 错过机会 |

### 五、分阶段建议
[立即做 / 评估后做 / 不做，附前提条件]

### 六、一句话总结
[用一句话概括核心判断]
```

## 关键规则

1. **先验证再推荐** — 声称「X 能做到 Y」前，必须验证 X 是否真的支持 Y
2. **用户环境优先** — 不是项目功能强就值得做，要看能不能在用户现有环境中跑起来
3. **分阶段不等于拖延** — 「暂不做」是清晰的决策，不是模糊的观望
4. **维护成本必须量化** — 不只是部署成本，还要算长期维护（依赖升级、配置变更、故障排查）
