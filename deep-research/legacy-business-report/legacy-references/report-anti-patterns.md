# 报告反模式（Report Anti-Patterns）

> **来源**：从 v1.0 → v5.1 实战反馈迭代而来。每一项反模式都对应一轮用户的具体反馈。
> **本文件目的**：让未来用本 skill 的人能直接读"什么不能做 + 为什么"，避免重蹈覆辙。

---

## 总览：8 大反模式

| # | 反模式 | 反馈轮次 | 影响等级 |
|---|--------|----------|----------|
| 1 | 文档中用 emoj | v4.0 → v5.0 | 🔴 严重（macOS Word 不渲染） |
| 2 | 彩色 callout 块 | v4.0 → v5.0 | 🟠 高（破坏专业感） |
| 3 | emoj 替代评级 | v4.0 → v5.0 | 🟠 高（信息密度低） |
| 4 | 报告结尾"报告结束"页 | v5.0 → v5.1 | 🟡 中（冗余） |
| 5 | 暴露过程态结果 | v5.0 → v5.1 | 🟠 高（读者视角错位） |
| 6 | 文件名做标题 | v2.0.0 → v3.0 | 🟡 中（标题应决策导向） |
| 7 | evidence_id 只放底部 | v2.0.0 → v4.0 | 🟠 高（验证效率低） |
| 8 | 多个独立工作文件 | v2.0.0 → v3.0 | 🟠 高（长期累积混乱） |

## 1. 文档中用 emoj（🔴 严重）

**反模式**：
```
⚠️ 关键判断：医疗是增速最快赛道，工业是最大单一细分市场。
💡 本章小结：工业外骨骼是受 4 类驱动的市场。
📌 结论：美欧 5% 渗透 $3.35B-$6.75B。
```

**为什么错**：
- macOS Word 中文字体不含 emoj glyph，**渲染为方框**（□）
- 截图证据：v4.0 报告中"⚠️ 关键判断"显示为"⚠️[方框]关键判断"
- 不仅是"丑"的问题——读者会怀疑文档是否被恶意修改

**正确做法**：
```
关键判断：医疗是增速最快赛道，工业是最大单一细分市场。
本章小结：工业外骨骼是受 4 类驱动的市场。
结论：美欧 5% 渗透 $3.35B-$6.75B。
```

**实现**：把所有 emoj 字符串替换为文字标签，python-docx 中不需要特殊处理。

## 2. 彩色 callout 块（🟠 高）

**反模式**（v4.0 样式）：
```
┌─────────────────────────────────────┐
│ [粉色背景 + 红色边框]                │  ← callout
│ 🔴 判断 1：医疗是增速最快赛道...     │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ [浅蓝背景 + 蓝色边框]                │
│ 💡 本章小结：工业外骨骼...           │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ [浅绿背景 + 绿色边框]                │
│ ✅ 核心建议：从无源被动式切入...     │
└─────────────────────────────────────┘
```

**为什么错**（用户原话："里面出现了很多的 emoji 和文本的错位问题，而且里面有非常多颜色的文本块儿这看起来很差"）：
- 麦肯锡风要求"克制"——彩色块显得花里胡哨
- 5 类不同颜色（粉/蓝/绿/黄/红）= 视觉噪音
- 读者注意力被色块吸引，而不是内容
- 印刷/复印时彩色背景变成灰底，更难看

**正确做法**（v5.0+ 麦肯锡风）：
```
判断 1. 医疗是增速最快赛道，工业是最大单一细分市场。
        （黑色正文，无背景色块）

本章小结：工业外骨骼是受 4 类驱动的市场。

核心建议：从无源被动式切入...
```

**实现**：
```python
def add_insight(doc, label, text):
    """关键判断 - 粗体段首 + 黑色正文（无背景）"""
    p = doc.add_paragraph()
    run = p.add_run(f'{label} ')  # 段首深蓝加粗
    run = p.add_run(text)         # 正文黑色
    # ❌ 不要加 pPr shd（背景色）
    # ❌ 不要加 pBdr（边框）
```

## 3. emoj 替代评级（🟠 高）

**反模式**（v4.0 表格里）：
```
| 驱动方       | 影响强度           |
|--------------|---------------------|
| 雇主 EHS     | ⭐⭐⭐⭐⭐（极强）   |
| 工会         | ⭐⭐⭐（强）         |
| 监管         | ⭐⭐（中）           |
| 保险         | ⭐⭐⭐（强）         |
```

**为什么错**：
- 同样的 emoj 渲染问题
- 信息密度低：5 个 ⭐ 才能表达"极强"，用文字 2 个字就够
- 表格对齐被 emoj 宽度干扰

**正确做法**：
```
| 驱动方       | 影响强度       |
|--------------|----------------|
| 雇主 EHS     | 极强（直接决策）|
| 工会         | 强（间接推动）  |
| 监管         | 中（软驱动）    |
| 保险         | 强（隐性驱动）  |
```

**例外**：如果坚持要视觉化评级，用"★"（一个字符）而不是"⭐"（emoj）。但仍推荐用文字。

## 4. 报告结尾"报告结束"页（🟡 中）

**反模式**（v5.0 之前）：
```
[page break]
[5 行空段]
————
[居中] 报 告 结 束
[居中] 本报告共 9 章正文 + 4 附录
[居中] Generated by Ryder AI Signal · 深度调研 · 2026-06-10
————
```

**为什么错**：
- 仪式感冗余——读者已经看完附录 D，自然知道报告结束
- "本报告共 9 章正文 + 4 附录"是元信息，对读者无价值
- "Generated by Ryder AI Signal"是制作方水印，不是报告内容
- 浪费一页纸（和报告正文一样重要的一页空间）

**正确做法**：报告在附录 D 自然结束，**不要再加分页 + 结束页**。

**实现**：
```python
# ❌ 反模式
doc.add_page_break()
for _ in range(5):
    doc.add_paragraph()
p = doc.add_paragraph('报 告 结 束')

# ✅ 正确做法
# 直接 return doc
return doc
```

## 5. 暴露过程态结果（🟠 高）

**反模式**（v5.0 附录 C）：
```
| URL                              | HTTP | Verdict                  |
|----------------------------------|------|--------------------------|
| germanbionic.com/                | 308  | unreachable（重定向）    |
| suitx.com/                       | 200  | reachable_no_keyword_hit |
| corporate.walmart.com/...        | 200  | reachable_no_keyword_hit |
| aboutamazon.com/...              | 404  | unreachable              |
| tescoplc.com/...                 | 403  | browser_required_or_blocked |
```

**为什么错**：
- `reachable_no_keyword_hit` 是**开发者视角**的脚本输出（verify_sources.py 的中间结果）
- 给读者看的是"这个 URL 怎样"，但读者关心的是"这个信源被采纳没"
- HTTP 状态码（308/200/404/403）对业务读者无意义
- 暴露过程态 = 把工具内部状态泄漏给最终用户

**正确做法**（v5.1 附录 C 重写为"信源透明度报告"）：
```
| 编号 | 信源                       | 类别 | 用途（对应正文 claim）                       | 状态     |
|------|----------------------------|------|----------------------------------------------|----------|
| S-01 | MarketsandMarkets 2025-2030| B    | 外骨骼总市场 $0.56B、CAGR 29.4%（第 1 章）   | 已采纳   |
| S-04 | SuitX 官网（IX 系列）      | C    | IX BACK VOLTON/AIR/SHOULDER 详细参数          | 已采纳   |
| S-12 | Reddit r/exoskeletons      |Social| 工人真实使用反馈（社区声音）                  | 备份     |
| S-18 | Tesco PLC People 数据      | C    | Tesco 员工数（仓储物流场景）                  | 不可达   |
```

**核心转换**：
- 视角：开发者 → 读者
- 维度：URL 状态 → 信源用途 + 采纳情况
- 状态描述：HTTP 码 → 业务三态（已采纳/备份/不可达）
- 关联：URL → 正文 claim（"用途"列直接说对应哪一章哪个数据）

## 6. 文件名做标题（🟡 中）

**反模式**：
```
文件: exoskeleton-global-labor-market-2026-06-10-v3.0.md
```

**为什么错**：
- 文件名 = 机器命名（主题+日期+版本号），不是"调研报告标题"
- 调研报告标题应"决策导向"——读者打开文件第一眼想知道"这份报告讲什么、结论是什么"
- 文件名含 v1/v2/v3 = 暴露迭代过程（用户并不关心你迭代了几次）
- 标题里出现日期版本号 = 像软件发布说明，不像研究报告

**正确做法**：
```
报告标题（Word 内）：国内外劳工外骨骼市场深度调研报告：玩家格局、商业模式与切入机会（2026 H1）
文件名（文件系统）：国内外劳工外骨骼市场深度调研报告-玩家格局商业模式与切入机会-2026-06-10.docx
```

**实现**：
- 报告标题 = `<主题>深度调研报告：<核心结论> (<窗口期>)`
- 文件名 = `<主题中文>-<核心结论简写>-<YYYY-MM-DD>.docx`
- 文件名**不含版本号**——迭代在标题内重写而非文件名累积

## 7. evidence_id 只放底部参考列表（🟠 高）

**反模式**（v2.0.0 风格）：
```
[正文]
德国 DGUV 是全球最严工伤预防体系之一。德国 BG 会主动为高风险岗位推荐
或补贴外骨骼采购。这是德国品牌（Ottobock Paexo、German Bionic）的
护城河。

...

[附录 D 参考来源]
26. DGUV - https://www.dguv.de/
27. OSHA - https://www.osha.gov/ergonomics
```

**为什么错**：
- 读者在正文遇到"DGUV"想验证来源时，必须翻到附录 D 第 26 条
- 翻页 + 找 URL + 复制 + 浏览器打开 = 4 步操作
- 大部分读者懒得做，导致"事实不验证"——失去 evidence_id 的核心价值

**正确做法**（v4.0+ 风格）：
```
[正文]
德国 DGUV：DGUV（德国法定工伤保险）是全球最严工伤预防体系之一。
德国 BG 会主动为高风险岗位推荐或补贴外骨骼采购。这是德国品牌
（Ottobock Paexo、German Bionic）的护城河。 [E26] DGUV  ← 可点击
                                                              ↑ 这里
```

**实现**（v4.0+）：
```python
def add_evidence_para(doc, lead_text, body_text, evidence_id, url, label=None):
    p = doc.add_paragraph()
    # 段首
    run = p.add_run(lead_text)
    run.font.color.rgb = COLOR_PRIMARY
    run.font.bold = True
    # 正文
    run = p.add_run(body_text)
    run.font.color.rgb = COLOR_BODY
    # 内联超链接（这是关键）
    if evidence_id and url:
        add_hyperlink(p, f'[{evidence_id}]', url, size=9.5)
```

**正文中 evidence_id 出现次数**：v3.0 = 0，v4.0 = 33 段（**几乎每个关键数据都有**），v5.0 = 30+ 段。

## 8. 多个独立工作文件（🟠 高）

**反模式**（v2.0.0 风格）：
```
~/Documents/Obsidian Vault/深度调研/
├── exoskeleton-global-labor-market-2026-06-10.md        (v1)
├── exoskeleton-global-labor-market-2026-06-10.md        (v2)
├── exoskeleton-global-labor-market-2026-06-10-v3.0.md   (v3)
├── research_plan.json                                   (5.6KB)
├── evidence_ledger.md                                   (5.4KB)
├── sources.csv                                          (2.1KB)
└── verified_sources.csv                                 (3.2KB)
```

**为什么错**（用户原话："我希望这个 skill 默认保存的文档是 word，而不是每次都产出那么多的文档，这个时间长了会造成很多的"）：
- 每个调研任务产生 4-5 个独立文件
- 长期累积：6 个月 = 20-30 个调研 × 5 文件 = 100-150 个文件
- 目录混乱：哪份是最终版？v1/v2/v3 都留着？
- 维护负担：plan/ledger 改了，3 个 .md 全文也要跟着改

**正确做法**（v3.0+ 风格）：
```
~/Documents/Obsidian Vault/深度调研/
├── 国内外劳工外骨骼市场深度调研报告-玩家格局商业模式与切入机会-2026-06-10.docx
│   （主交付物，单一文件 143KB，含 4 附录）
└── 国内外劳工外骨骼市场深度调研报告-2026-06-10.md
    （Obsidian 索引，2.3KB，5 条核心判断 + 链接到 .docx）

.archive/  （历史归档，仅用于对照）
├── exoskeleton-global-labor-market-2026-06-10.md
├── exoskeleton-global-labor-market-2026-06-10-v3.0.md
└── 国内外劳工市场外骨骼产品调研.md
```

**实现**：
- plan/ledger/sources/verified → 嵌入 Word 附录 A-D（不产生独立文件）
- 工作文件 → `/tmp/<项目名>/`，交付后清理
- Obsidian 目录 → 主目录只有 1 个 .docx + 1 个 .md 索引
- 历史归档 → `.archive/` 子目录（按需访问）
