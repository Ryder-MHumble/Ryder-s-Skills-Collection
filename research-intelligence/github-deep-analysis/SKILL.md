# GitHub 开源项目深度分析

## 体裁定义

本文档生成的是「有洞察的可验证判断报告」。

三个词缺一不可：
- **可验证**：每个核心判断都有证据链，读者可以沿着链接验证
- **有洞察**：有让人「woc 没想到」的另类角度
- **判断报告**：读者读完持有一个更精确的判断，不是「涨知识了」

---

## 触发条件

用户发送 GitHub 仓库链接（github.com/xxx/xxx），且意图是了解/分析该项目。

**🚨 认知层优先：拿到链接后，必须首先执行升维检查，确认用户真实意图。技能触发不等于认知闭环，必须在执行预设流程前复查"这个预设路径是否匹配用户真实意图"。**

### 意图信号优先级排序（从高到低）

| 意图信号 | 分析路径 | 升维解析 | 信号权重 |
|---------|---------|----------|----------|
| 「当前市面上这么多 X，这个有啥不一样」「和 XX 相比如何」「对比分析」 | **对比辨析框架** | 用户需要竞品定位分析，在已有生态中识别差异化价值点 | ⭐⭐⭐⭐⭐（最高） |
| 「值不值得做」「能不能集成」「要不要用」 | 战略决策简报（快速可行性评估） | 用户需要决策输入，关注可行性、成本、风险 | ⭐⭐⭐⭐ |
| 「技术怎么样」「代码质量」「架构」 | 技术评估（简短） | 用户需要工程/架构层面的专业判断 | ⭐⭐⭐ |
| 「分析一下」「写篇报告」「这个怎么样」 | 全维度深度分析 | 用户需要完整的战略判断，关注项目的长期价值和赌注 | ⭐⭐ |

**对比辨析框架激活条件（强制触发）**：
- 用户明确提及"市面上已有工具"、"相比其他"、"对比"、"有什么不同"、"差异"等关键词 → **立即激活对比分析路径**
- 或用户在描述中暗示需要差异点识别（如"这个有啥不一样"、"和现有方案比"） → **立即激活对比分析路径**
- 当对比信号与其他信号共存时，**对比信号权重最高**，优先响应对比意图

### 认知层保护检查清单（技能触发后立即执行）

**⚠️ 历史教训：2026-05-14 CloakBrowser分析请求中，用户问"市面上这么多的浏览器自动化工具有啥不一样"，明确包含对比信号。但技能触发后注意力转移到预设流程，未先确认用户真实意图，导致在错误方向上高效执行。**

检查步骤：
1. **信号识别**：用户话语中是否有对比/比较/选择信号？
2. **权重评估**：对比信号 vs 模糊分析信号，哪个更强？
3. **意图澄清**：如果信号有歧义，必须主动确认："您需要的是全面战略分析，还是与其他工具的对比定位分析？"
4. **路径选择**：根据最高权重信号选择分析框架

**关键规则**：
- 对比信号权重 > 模糊分析信号
- 技能触发≠认知闭环，必须保持升维优先
- 宁可多问一句确认，不要默认选择预设路径
- 对比分析可作为深度分析的补充章节，也可作为独立报告

**⚠️ 关键认知层检查（技能触发惯性防范）**：
当匹配上述激活条件时，必须执行以下升维检查：
1. **确认信号权重**：对比信号（"有啥不一样"、"相比其他"、"对比"）比"深度分析"、"分析一下"等模糊信号更强，优先响应对比意图
2. **技能惯性自检**：在加载github-deep-analysis技能后，立即检查是否会默认走全维度战略分析路径。技能触发会制造"认知已闭环"的假象，必须在执行预设流程前复查"这个预设路径是否匹配用户真实意图"
3. **即时澄清**：如用户对比意图明确但仍有歧义，主动问"您需要的是全面战略分析，还是与其他工具的对比定位分析？"
4. **双框架兼容**：如用户同时需要深度和对比，优先保证深度分析质量，在结论部分补充对比维度

**🚨 历史教训**：2026-05-14 CloakBrowser分析请求中，用户问"当前市面上这么多的浏览器自动化工具，这个有啥不一样"，明确包含对比信号。但github-deep-analysis技能被触发后，注意力转移到技能预设流程，直接走了全维度战略分析路径，未先确认用户真实意图是比较不同工具的核心差异还是全面战略分析。这是典型的"技能触发认知层失效"问题。必须避免。

**核心陷阱防范**：Skill触发制造"认知已闭环"的假象，注意力从"这是不是正确的问题"转移到"如何执行skill预设流程"。必须在执行预设流程前复查"这个预设路径是否匹配用户真实意图"。

### 对比辨析分析路径（详细指南可参考 `references/contrast-analysis-framework.md`）：
1. **竞品生态定位** — 快速扫描同类项目列表（GitHub搜索、GitHub Topic），建立竞品矩阵
2. **差异化核心** — 聚焦项目在技术栈、商业模式、用户体验、部署方式上的独特之处
3. **白空间识别** — 这个项目填补了竞品矩阵中的哪个空白？
4. **替代性分析** — 在什么场景下用户会选择这个而不是其他工具？
5. **可持续差异化** — 这种差异化是短期技术优势还是长期结构性优势？

**关键升维检查**：
- 当用户提及"市面上有很多工具"时，必须主动确认：用户需要的是深度分析还是对比定位？
- 对比分析不能代替深度分析，但可以作为深度分析的补充视角
- 如果用户同时需要深度和对比，优先保证深度分析质量，在结论部分补充对比维度

## 快速可行性评估模式

当用户意图是「值不值得做/能不能集成」时，跳过全维度深度分析，走快速评估路径：

### 评估流程

1. **项目本质澄清** — 这是什么？依赖什么 runtime？核心功能？部署方式？
2. **环境兼容性** — Runtime / 调用链路 / 状态同步 / 维护成本 四维评估 → 可行性评级 ⭐~⭐⭐⭐⭐⭐
3. **场景匹配** — 项目功能 vs 用户痛点 vs 匹配度矩阵
4. **方案建议** — A.直接集成 / B.借鉴理念 / C.暂不做（附成本与风险）

### 输出格式（简报）

```markdown
## 评估结论：[直接集成 / 借鉴理念 / 暂不建议]

### 项目本质
[一句话定义 + 核心功能 + 依赖 runtime]

### 兼容性
[Runtime / 调用链路 / 状态同步 / 维护成本]

### 方案对比
| 方案 | 可行性 | 成本 | 风险 |
|------|--------|------|------|

### 分阶段建议
[立即做 / 评估后做 / 不做，附前提条件]
```

### 关键规则
- **先验证再推荐** — 声称「X 能做到 Y」前必须验证
- **用户环境优先** — 不是功能强就值得做，看能不能在用户环境中跑
- **维护成本必须量化** — 不只部署成本，还要算长期维护

**如果用户只发了链接没有任何说明，默认走全维度深度分析。**

## 全维度深度分析的写作原则

### 核心：你不是在评估技术，你是在判断一个赌注

每个开源项目背后都有一群人押了某个赌注——押对了就是赛道赢家，押错了就是昙花一现。

你的任务是：找到这个赌注是什么，判断它押对了还是押错了，押对的条件下限是什么，押错的上限是什么。

赌注不一定在 README 里明说，有时候藏在数据反常里，有时候藏在团队背景里，有时候藏在发布时间线里。

### 分析维度由项目的主要矛盾决定

项目可能涉及以下维度：

| 维度 | 问的核心问题 |
|------|------------|
| 技术 | 这个项目的核心工程难题是什么，解决得怎么样 |
| 产业 | 这个项目在产业链里占什么位置，谁会为它付费 |
| 商业 | 这个项目的成本结构和收益模式是什么，可持续吗 |
| 市场 | 这个项目的目标用户是谁，有没有被满足的需求 |
| 人伦 | 这个项目有没有伦理风险，用户会不会误用 |

正文里各维度比例由数据决定——如果某个维度信息密度高就展开，另一个维度数据空洞就一笔带过。不是说「本篇分析技术、市场、商业三个维度」，而是「这个项目的主要矛盾在技术层，第二层矛盾在商业层』。

### 写作的灵活性原则

**框架是帮我内部思考的工具，不是输出的模板。**

分析阶段：可以用框架引导思考，问自己「这个项目的主要矛盾是什么」「哪些维度信息密度高」「数据的反常在哪里」。

写作阶段：跟着数据走。发现了一个反常的数据矛盾，就顺着它挖；某个维度展开后信息量大，就多写；另一个维度数据空洞，就一笔带过。不要为了「完整覆盖所有维度」而硬凑字数。

4 篇参考文档的实际结构（用来校对我对「灵活写作」的理解是否正确）：
- MemPalace：技术（96.6% 数据拆解）+ 商业（存储成本可持续性）+ 叙事伦理（营销夸大 vs 真实发现）
- OpenCLI：技术（CDP架构）+ 市场（协议真空期套利）+ 商业（成本形态转移）+ 安全伦理
- nuwa-skill：技术（prompt工程 vs 真蒸馏）+ 市场（名人思维矿石）+ 人伦（「像」和「是」的陷阱）
- Archon：技术（YAML DAG）+ 市场（空白右上角象限）+ 商业（workflow as 团队资产）+ 产业（窗口期判断）

注意：这 4 篇没有一篇是「先定维度再填内容」的结构。都是核心判断 → 顺着判断挖数据 → 数据带出真实维度 → 维度矛盾自然形成章节。

## 分析流程

### Phase 0：快速扫描，找到主变量

在采集数据之前，快速扫 README 和 GitHub 首页，问自己：

**这个项目的命运最关键地取决于什么？**

把主变量写下来。它必须是一个有立场、有判断的 POV，不是描述性说明。

- 差的主变量：「这是一个 WiFi 姿态估计项目」——描述性，没有判断方向
- 好的主变量：「ESP32-S3 在真实家居环境中的稳定性，决定了这个项目是革新还是玩具」——有立场，有边界条件

**反向压力测试**：如果我的判断是反的，有什么证据支持？列不出来说明可能是预设立场，不是分析出来的洞察。

### Phase 0.5：发现反常

在扫描数据时，关注「不协调」的地方：

| 反常类型 | 什么情况 |
|---------|---------|
| 数据矛盾 | star 高但 commit 少、fork 多但 issues 冷清 |
| 声称 vs 实际 | README 说「全自动」但核心逻辑是手动配置 |
| 叙事 vs 工程 | 包装精致但实现简单、架构图复杂但代码少 |
| 热度 vs 质量 | star 增速异常快但没有 KOL 推文佐证 |
| 时间线反常 | 创建时间 vs star 爆发时间不匹配 |
| 团队 vs 产品 | 团队背景暗示的方向 vs 产品实际走的路 |

**要求**：列出 3+ 个具体反常，每个必须有数据支撑。

**关键**：反常是洞察的种子，不是装饰品。后续章节要回应这些反常——解释它们或者验证它们不成立。

### Phase 1：数据采集

采集方式（控制成本，不 clone 源码）：
- `curl -sL https://raw.githubusercontent.com/{user}/{repo}/main/README.md` 获取 README
- `curl -sL https://api.github.com/repos/{user}/{repo}` 获取精确元数据（stars、forks、创建时间、更新时间）
- `curl -sL "https://api.github.com/repos/{user}/{repo}/issues?state=open&per_page=20&sort=comments"` 获取社区反馈（按评论数排序，抓最热门 issues）
- `curl -sL "https://api.github.com/repos/{user}/{repo}/commits?per_page=30&sort=date"` 获取最近 commit
- 浏览器打开项目官网（如有），看产品定位和目标用户描述

Twitter KOL 采集（如需要）：使用 twitterapi.io 的 `tweet_advanced_search` 端点，Header 认证 `x-api-key: $(grep TWITTERAPI_IO_KEY ~/.hermes/.env | cut -d= -f2)`。

### Phase 2：脑子里形成洞察

读数据时，让反常和主变量在脑子里碰撞，形成 2-3 个核心洞察。

**洞察的特征**：
- 不是从框架里选出来的，是数据告诉我的
- 有具体的证据支撑
- 能让读者说「这个角度我没想到」

**竞争坐标系**（仅当它能揭示新东西时才写）：
- 找到分类轴的本质原因，而不是罗列显而易见的差异
- 白空间识别：这个位置为什么之前没人占？
- 如果坐标系不能揭示新东西，就不写

## 输出质量标准

### 硬限制（零容忍）

1. **无分割线**：`add_divider()` 禁止使用
2. **真实 references**：每个关键 claim 附来源 URL，不能编造
3. **不堆 emoji**：适量点缀可以，整段都是 emoji 就删
4. **纯文本段落**：飞书不支持 Markdown 渲染，`add_p()` 传入纯文本
5. **返回文档链接**：写完后必须直接给出飞书文档 URL

### 判断前置

文章前 200 字内必须给出最核心的判断。

不允许用故事铺垫判断。叙事性元素可以在判断之后作为论据出现。

### 论据闭环

每个核心论点必须完成三步：
1. **断言**：我的判断是什么
2. **证据**：支撑这个判断的具体数据（数字、链接、社区反馈）
3. **推论**：如果这个证据成立，结论是什么

不允许：断言 → 类比 → 另一个断言。

### 段落递进

每个段落结束时，读者应该比读这段之前更接近核心判断。不是「多知道了一些信息」，而是「离判断更近了一步」。

### 叙事弧线

有推进感，章节之间有逻辑递进，不是平行罗列。

好：第二章建立在第一章的判断上，第三章回应第二章发现的新问题。
差：四个章节各自独立，读完任何一个不影响对其他的理解。

### 可验证

每个关键 claim 有来源，读者可以追溯验证。

## 5个逻辑自检

写完每一章后、发布前，用这5个问题过一遍。任何一条回答「是」都要改。

**问题 1：这个判断有来源吗？**
没有 → 换论证方式，不能无来源下判断。

**问题 2：读者能自己验证吗？**
不能 → 标清「推断」还是「事实」，不能把推断当事实写。

**问题 3：这个 claim 有其他合理解读吗？**
有 → 补全解读，说「目前证据最支持第 X 种」。不能只论证一种然后暗示其他都错。

**问题 4：我用了暗示恶意的词吗？**
用了（「压制」「诈骗」「grifter」）→ 改成中性描述，不能用情绪标签代替事实。

**问题 5：这篇是「框架执行」还是「真正看懂了一个项目」？**
如果删掉所有框架标注，文章本身还有价值吗？
没有 → 删掉这些装饰，写真正来自数据的洞察。

## 文章结构

> 章节数量和顺序由洞察驱动，不是由模板决定。

开头结构是固定的，但具体内容由洞察决定：
1. 最核心的判断——直接给结论，包含赌注描述
2. 关键前提——这个判断在什么条件下成立
3. 失效条件——如果前提不成立，判断怎么变

章节由项目的主要矛盾决定。标题必须：
- 有编号（一、二、三...）
- 从项目本身提炼，不用「技术架构」「竞争分析」这种维度名
- 标题暗示判断方向，不只是描述内容

收尾包含：
1. 核心判断重述（2句话）
2. 判断失效条件
3. 可操作的下一步（如果适用）

Further Reading 最低 3 条，每条包含：真实链接 + 为什么与核心判断直接相关 + 读者应该带什么问题去读。

### FeishuWriter API 参考

| 方法 | 用途 |
|------|------|
| `FeishuWriter(title)` | 创建写入器实例 |
| `add_h1(text)` | 一级标题 |
| `add_h2(text)` | 二级标题 |
| `add_h3(text)` | 三级标题 |
| `add_p(text, links=None)` | 段落（纯文本，不加 Markdown）；links 为 `{"显示文本": "url"}` |
| `add_quote(text, links=None)` | 引用块 |
| `add_bullet_list(items, links_per_item=None)` | 无序列表（用「•」前缀模拟） |
| `add_numbered_list(items, links_per_item=None)` | 有序列表（用数字前缀模拟） |
| `add_table_from_data(headers, rows)` | 简单表格（用段落模拟） |
| `publish()` | 创建文档并写入，返回 `{doc_id, doc_url, block_count}` |
| `attach_to_wiki(space_id, parent_node_token)` | 挂载到 wiki 节点（可选） |

**常见坑：**
- `publish()` 不接受 wiki 参数，wiki 挂载是独立的 `attach_to_wiki()` 方法
- 不要从 `feishu_writer` 模块导入 `add_h1` 等函数（它们是类方法，不是模块级函数）
- import 路径是 `research/github-deep-analysis/scripts`，不是 `productivity/feishu-docs/scripts`

### 已知兼容问题
- `add_p()` 传入纯文本，不支持 Markdown 渲染
- link 必须放在 `text_element_style` 内部
- **Python heredoc 执行注意**：中文字符串内的中文引号（"" ''）会导致 SyntaxError。解决方案：用 `"""..."""` 包裹含中文引号的字符串，或直接移除中文引号。绝不用 write_file + terminal 执行含中文引号的 heredoc，用 execute_code 或 `python3 << 'PYEOF'` 的 heredoc 模式。
