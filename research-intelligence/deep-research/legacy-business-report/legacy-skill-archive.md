# Deep Research Report

把调研从“报告模板”升级为“可追溯的证据系统”。核心原则：**先写检索计划，再搜；先建证据账本，再下判断；每个关键结论都有 evidence_id；URL 可达不等于证据成立。**

## 硬性规则

- **无计划不检索**：动手前先写一版检索计划，明确目标、范围、时间窗、地域、语言、排除项、source mix、停止条件。
- **无证据不写入**：所有数据、排名、趋势、竞品判断必须绑定 `evidence_id`；找不到可靠来源时写“未公开/未验证”，不要补脑。
- **先验范围优先**：用户限定国家、场景、品类、时间窗时，不自行扩展；扩展建议只放“可选下一步”。
- **近况必须实时搜**：涉及“最新、近期、近30天、趋势、舆情、用户怎么说、竞品最近动作”时，必须走近30天/社媒信号路径。
- **HTTP 只算门槛**：交付前必须验证 URL 可达；关键来源还要验证标题/正文是否真的支持该 claim。
- **交付前反幻觉审计**：扫描“第一、唯一、最大、最快、显著、市场规模、占比、融资、估值、用户量”等高风险断言，缺 evidence_id 就删除或降级。
- **交付前反幻觉审计**：扫描"第一、唯一、最大、最快、显著、市场规模、占比、融资、估值、用户量"等高风险断言，缺 evidence_id 就删除或降级。

## 视觉样式系统（麦肯锡风）

**本 skill 默认输出麦肯锡风格的商业咨询报告。以下规则在 5+ 轮用户反馈后固化，不可违反。**

### 配色（2 色收敛）

| 用途 | 颜色 | 用法 |
|------|------|------|
| 主色 | 深蓝 `#1F4E79` | 标题、表头、链接、关键强调 |
| 辅色 | 中性灰 `#7F7F7F` | 次要文字、辅助说明、注释 |
| 表格交替行 | 浅灰 `#F2F2F2` | 数据行底色 |
| 关键数字 | 深红 `#C00000` 或深蓝 | 16-18pt 粗体大字，**仅用于数字本身** |
| 背景 | 纯白 `#FFFFFF` | 页面与表格主底色 |

**禁止**：第三种主色、彩色背景块、彩虹色表头、emoj 装饰色。

### 排版元素规则

- **关键判断**：粗体段首（深蓝）+ 黑色正文（**无背景色块**）
- **关键数字**：16-18pt 粗体大字 + 深蓝或黑色
- **表格**：白底 + 蓝头条（深蓝 + 白字加粗）+ 灰交替行
- **章节封面**：1 条细蓝横线 + 章节号（小灰字）+ 大标题（22pt 蓝色加粗）
- **行距**：1.4-1.6
- **字体**：SimSun（中文）
- **Banner 图**：matplotlib 渲染深蓝科技感封面（渐变 + 抽象数据线 + 关键数字）

### 严禁的反模式（Hard DON'Ts）

| 反模式 | 反馈来源 | 替代方案 |
|--------|----------|----------|
| ❌ 文档中用 emoj（🔴💡📌⚠️🎯） | macOS Word 中文系统不渲染 emoj glyph，显示方框 | 用文字"关键判断："、"建议："、"小结：" |
| ❌ 彩色 callout 块（粉/蓝/绿/红/黄背景） | "颜色文本块儿这看起来很差" | 粗体段首 + 黑色正文（无背景） |
| ❌ emoj 替代评级（⭐⭐⭐⭐⭐） | 渲染错位 + 信息密度低 | 用文字"极强 / 强 / 中 / 弱" |
| ❌ 报告结尾加"报告结束 / 本报告共...章"页 | 仪式感冗余 | 报告在附录 D 自然结束 |
| ❌ 暴露过程态结果（"reachable_no_keyword_hit" / HTTP 状态码） | 给读者看的是结论不是过程 | 用"已采纳 / 备份 / 不可达"三态 |
| ❌ 文件名做标题（`xxx-report-2026-06-10-v3.0`） | 标题应"决策导向"不是机器命名 | 标题 = `<主题>+<核心结论>+<窗口期>` |
| ❌ evidence_id 只放底部参考列表 | 读者验证来源需翻页 | **正文每个 evidence_id 内联可点击** |
| ❌ 拆成 plan/ledger/sources 多个独立文件 | 长期累积混乱 | 嵌入 Word 附录 A-D（单一 .docx） |

### 必含元素

- ✅ 封面 Banner 图（matplotlib 渲染）
- ✅ 单一 .docx 交付（用户显式要 .md 时例外）
- ✅ 30+ evidence_id 全程绑定关键判断
- ✅ 5W2H 主动反问（用户给宽泛关键词时）
- ✅ 报告标题与文件名分离
- ✅ 附录 C 为"信源透明度报告"（不是"机械预筛输出"）

详细技术实现见 `references/mckinsey-style-guide.md` 和 `references/report-anti-patterns.md`。

## 工作流

### 1. 升维确认（5W2H 主动反问）

**核心原则**：当用户给"宽泛主题"或"几个关键词"时，**必须主动反问补齐 5W2H context**，再开始检索。这避免跑出"看似全面但偏离真实需求"的报告。

#### 1.1 何时反问

- 用户只给 1-2 个宽泛词（如"AI Agent 调研"、"工业外骨骼"、"机器人市场"）
- 用户没说明决策目的（投资 / 竞品对比 / 产品立项 / 兴趣探索）
- 用户没说明受众（自己 / 团队 / 领导 / 投资人 / 客户）
- 用户没说明地域边界（默认全球，但用户可能只想看国内或某个国家）
- 用户没说明时间窗（默认最近 12 个月，但有时是 3-5 年趋势）

直接给出**反问清单**（不要直接动手搜）：

```
为了把这个调研做到最贴近你真实需求，我需要先对齐几个关键点：

1. 决策目的：这份报告主要用途是什么？
   - 投资判断（VC/PE/二级市场）
   - 产品立项（你自己或团队做这个方向）
   - 竞品对比（已有产品，对标对手）
   - 内部汇报（给领导/团队看）
   - 兴趣探索（个人了解行业）
   - 其他：____

2. 受众与交付物：
   - 谁会读这份报告？你自己 / 团队 / 领导 / 投资人 / 客户 / 其他？
   - 报告长度预期：一页纸 / 5-10 页 / 20+ 页深度 / 不限？
   - 输出格式：Word / PDF / PPT / Excel / 不指定（推荐 Word）？

3. 地域边界：
   - 全球 / 中国 / 美国 / 欧洲 / 特定国家 / 不限？
   - 是否包含海外市场（如国内看海外机会）？

4. 品类/场景边界：
   - 工业 To B / 消费 To C / 医疗 / 金融 / 教育 / 其他？
   - 包含哪些场景 / 排除哪些场景（如不要消费级、不要医疗康复）？

5. 时间窗：
   - 最近 30 天 / 最近 3 个月 / 最近 12 个月 / 3-5 年趋势 / 不限？

6. 重点关注维度（可多选）：
   - 市场容量与增速
   - 主要玩家与竞争格局
   - 技术路线与产品对比
   - 客户画像与需求
   - 商业模式与定价
   - 法规与合规
   - 投资 / 融资 / 并购动态
   - 行业进入壁垒与机会点

7. 已有材料（可选）：
   - 你已经有任何参考资料、报告、数据、品牌名单吗？优先作为候选池。

回复后我会基于这些 context 写一份 research_plan.json 并开始检索。
```

#### 1.2 何时不反问（直接动手）

- 用户已明确给出 5W2H 中至少 5 项（如"我想做工业外骨骼劳工保护美国市场调研，目的是投资判断，重点看玩家格局和商业模式"）
- 用户在已有报告上追问（如"再深挖一下 German Bionic 的客户案例"）
- 用户要求"快速扫描"或"先给我看一版"（这时按默认假设走：全球 / 工业 To B / 最近 12 个月 / Word 输出）

#### 1.3 默认假设（不反问时使用）

如果用户没反问，**声明这些默认假设**再开始：

- **决策目的**：商业决策（最常见）
- **受众**：自己 + 团队
- **地域**：全球（含中国）
- **品类/场景**：按用户提到的第一关键词推断
- **时间窗**：最近 12 个月
- **输出格式**：单一 .docx 文件
- **重点维度**：市场容量 + 玩家格局 + 商业模式（麦肯锡"三议题"格式）

#### 1.4 反问不是"形式主义"

反问的目的是**让用户参与定义问题**，不是为了"显得专业"。如果用户嫌烦，可以一句话回答（"投资判断、Word、全球、工业 To B、最近 1 年"）就足够。

反问的最大价值：避免跑出"形式完整但内容空泛"的报告。

### 2. 写检索计划

在第一次检索前，用 5-12 行列出计划。复杂任务写 JSON 草案；详细 schema 见 `references/deep-research-search-protocol.md`。

最小计划包含：

| 字段 | 要求 |
|------|------|
| Research question | 报告要回答的决策问题，不只是主题词 |
| Scope | 地域、时间窗、品类、排除项 |
| Source mix | A/B/C 源、社媒/社区、公司官网、数据库、学术/政策 |
| Subqueries | 4-8 个独立检索问题，覆盖市场、玩家、客户、政策、负面证据 |
| Entity resolution | 公司官网、产品名别名、创始人/账号、GitHub repo、subreddit/community |
| Stop condition | 何时停止继续搜：关键结论被交叉验证、或公开信息不足并已标注边界 |

### 3. 选择检索路径

- **市场/行业全景**：先 4-6 个宽面查询建立候选池，再 4-6 个定向验证重点公司/数据。
- **竞品/Top N 对比**：先锁定候选名单，再逐公司验证官网、产品、客户、融资、价格、负面信息。
- **近30天趋势/用户声音**：优先使用 last30days 能力或同等手工路径，覆盖 Reddit/X/YouTube/TikTok/HN/GitHub/Polymarket/Web；社媒信号用于“用户怎么说/热度/争议”，不能单独证明市场规模。
- **中国机构/政策/产业**：优先人民网、第一财经、官方公众号/官网、政府公文、公司官网；工商数据受限时向用户要材料。
- **学术/海外机构**：优先 OpenAlex、Crossref、论文 PDF、机构官网、政府/监管文件。
- **外部报告评估**：先标记“好/存疑/坏”，再验证来源并吸纳结构价值，不只挑错。

### 4. 建证据账本

每轮检索后先更新证据账本，再写报告。账本是**工作文件**（不是独立交付物），最终会嵌入 Word 文档的"附录 B：证据账本"。文件格式可用 Markdown 表格、CSV 或 JSONL。

| evidence_id | claim | source | type | date | supports | confidence | note |
|-------------|-------|--------|------|------|----------|------------|------|
| E01 | 具体可被验证的断言 | 标题 + URL | A/B/C/Social | 发布/访问日期 | yes/partial/no | high/medium/low | 摘要、关键词、限制 |

证据分级：
- **A**：政府/监管/SEC/财报/公司 IR/正式统计/论文原文。
- **B**：权威媒体、行业研究机构、可信数据库、严肃垂直媒体。
- **C**：公司官网、产品页、招聘页、会议材料、一般行业资讯。
- **Social**：Reddit/X/YouTube/TikTok/HN/GitHub/Polymarket/App Store 等实时信号。

强度规则：
- 关键量化数据：优先 1 个 A/B 源；没有时至少 2 个独立 C 源，并标“估算/口径”。
- 重大判断：至少 2 条独立证据，且包含一个反向检索或负面证据检查。
- 社媒结论：写互动量、时间窗、样本偏差；不要把热帖当总体市场结论。

### 5. 验证 source

交付前必须做两层验证：

1. **可达性验证**：HTTP 200/301/302 可保留；403 只有在浏览器确认内容且标注“需浏览器访问”时可保留；404/000 不写入。
2. **内容命中验证**：标题/正文必须出现公司名、产品名、数据关键词或同义表达；只可达但不支持 claim 的链接降为 background 或删除。

可用脚本：

```bash
python3 scripts/verify_sources.py sources.csv --timeout 12 --out verified_sources.csv
```

CSV 最少包含 `url,keywords`；`keywords` 用分号分隔。脚本只能做机械预筛，最终 claim 是否成立仍要人工判断。

**重要**：v5.1+ **禁止**在最终报告附录 C 直接展示"reachable_no_keyword_hit"等机械预筛术语。详见 `references/report-deliverable-conventions.md` § 6.10（附录 C 是"信源透明度报告"）。

### 6. 综合与写作

默认结论先行：

1. **核心判断**：3-5 条，每条带 `evidence_id` 和 WHY。
2. **范围与方法**：一句话说明时间窗、地域、source mix、限制。
3. **事实层**：市场、玩家、政策、技术、用户声音，按维度组织，不按搜索来源散列。
4. **对比层**：竞品表、证据强度、置信度、数据缺口。
5. **建议层**：机会、风险、切入点、下一步验证。
6. **证据与边界**：来源表、不可得数据、估算口径、删除/降级的信息。

工业硬件/To B 产品必须额外覆盖：概念厘清、谁买单、价格/ROI、法规驱动、渠道、部署/客户案例、竞争空白。详细旧版方法和案例见 `references/research-report-legacy-playbook.md`、`references/exoskeleton-labor-protection-market.md`。

### 6.1 交付物格式（默认 Word 单一文件 + 麦肯锡风）

**默认输出**：单一 `.docx` 报告（v5.1+ 麦肯锡商业咨询风格）。用户显式要 `.md` / `.pdf` 时例外。

**为什么是 Word**：
- 用户常需编辑/调整结论（vs PDF 只读）
- 可点击超链接（30+ evidence_id 全部内联可点击跳转）
- 复杂表格样式（KV 表格 / 交替行底色 / 关键数字突出）
- 文件小（50-150KB vs PDF 200-500KB）
- 中文字体可控（SimSun + East Asia 字符集）

**v5.1+ 视觉规范（麦肯锡风，硬性）**：

| 维度 | 规范 |
|------|------|
| 配色 | 2 色：深蓝 `#1F4E79` + 中性灰 `#7F7F7F` + 黑白 |
| 表格 | 表头深蓝底白字、数据行白底/浅灰交替、左列加粗深蓝 |
| 章节封面 | 细蓝线 + 章节号 + 标题 + 副标题（不要大色块/emoji） |
| 关键数据 | 粗体 16-18pt + 深蓝 + 上方小灰字标签 + 下方可点击超链接来源 |
| 段落 | 行距 1.5 / 段间距 4pt / 首行不缩进 |
| 文字 | 中文 SimSun + 英文默认（避免中英混排字距问题） |
| 首页 | matplotlib 渲染 900x300px 深蓝科技感 Banner 图 |

**v5.1+ 视觉禁令**：

- ❌ **任何 emoji**（macOS Word 中文字体不含 emoji glyph，渲染为方框 □）
- ❌ **彩色 callout 块**（浅红/浅蓝/浅绿/浅黄底色 + 红色/彩色边框）
- ❌ **附录 C 暴露 verify_sources.py 过程态**（reachable_no_keyword_hit 等机械预筛术语）
- ❌ **结尾页**（"报告结束 / 本报告共 N 章 / Generated by ..."）
- ❌ 多色表格行（红/绿/黄交替）

**替代方案**：
- ❌ 彩色 callout → `add_insight(doc, label, text)` 粗体段首 + 黑色正文
- ❌ emoji → 文字（"关键判断：" / "高/中/低" / "极强/强/中/弱"）
- ❌ 附录 C 过程态 → "信源透明度报告"（编号 / 信源 / 类别 / 用途 / 状态）
- ❌ 结尾页 → 报告在附录 D 自然结束

详细视觉规范、颜色常量、`add_insight` / `render_banner` 实现、附录 C 模板见 `references/report-deliverable-conventions.md` § 6 + `templates/research_report_skeleton.py`（v5.1）。

**工作流产物处理（避免"多文件累积"）**：
- `research_plan.json` / `evidence_ledger.md` / `sources.csv` / `verified_sources.csv` —— **不作为独立交付物**
- **嵌入 Word 文档的附录**：
  - 附录 A：检索计划（research_plan.json 转 KV 表格）
  - 附录 B：证据账本（evidence_ledger.md 转多列表格）
  - 附录 C：**信源透明度报告**（编号 / 信源 / 类别 / 用途 / 状态，**不**暴露 verify_sources.py 过程态）
  - 附录 D：参考来源（30 evidence_id URL 列表，含可点击超链接）
- 审计能力完整保留，但用户打印/分享时只打印正文 § 1-6，附录按需启用

**Obsidian 目录管理（避免"v1.md / v2.md / v3.md"累积）**：
- 主交付物：单一 `.docx`（按 § 6.2 命名）
- 可保留：`.md` 索引文件（标题 + 链接到 `.docx` 路径 + 一句话结论）
- 不再保留：每轮迭代的 `.md` 全文 / `.json` 检索计划 / `.csv` 来源表
- 工作文件（plan/ledger/sources/verified）放 `/tmp/<项目名>/`，交付后清理

### 6.2 标题与文件名分离

调研报告的标题和文件名是**两件事**——标题面向读者（决策导向），文件名面向文件系统（机器友好）。把它们混在一起是新手常见错误。

**报告标题**（Word 文档内 + 文档第一行）：
- 格式：`<主题>深度调研报告：<核心结论> (<窗口期>)`
- 例：`国内外劳工外骨骼市场深度调研报告：玩家格局、商业模式与切入机会（2026 H1）`
- 反映"决策导向"，可给领导/投资人直接看
- 副标题用括号说明时间窗或场景

**文件名**（文件系统标识）：
- 格式：`<主题中文>-<核心结论简写>-<YYYY-MM-DD>.docx`
- 例：`国内外劳工外骨骼市场深度调研报告-玩家格局商业模式与切入机会-2026-06-10.docx`
- **不含版本号**（v1/v2/v3）——迭代在标题内重写而非文件名累积
- 主题词用破折号分隔核心结论简写，方便文件系统搜索

**为什么不用日期做版本号**：
- 同一份调研报告不会同一天做两次，日期就是时间戳
- 真实迭代是"标题改写"（"工业外骨骼市场调研"→"外骨骼劳工保护市场调研：玩家格局"），不是"v1→v2→v3"

### 7. 交付前审计

逐项确认：

- [ ] 每个核心判断后都有 `evidence_id`。
- [ ] 每个 URL 已验证可达，关键来源已验证内容命中。
- [ ] 最新/趋势/用户声音任务已覆盖近30天或明确说明无法覆盖。
- [ ] 社媒信号没有被当作官方事实来源。
- [ ] 估算值写明口径、公式、假设和置信度。
- [ ] 用户限定范围没有被自行扩大。
- [ ] 报告没有"搜索工具受限"借口，只写数据边界。
- [ ] **报告以单一 `.docx` 交付，`research_plan.json` / `evidence_ledger.md` / `sources.csv` / `verified_sources.csv` 已嵌入 Word 附录 A-D，未作为独立文件产生**。
- [ ] **报告标题是"决策导向"调研报告标题（主题+核心结论+窗口期），不是文件名式（不含日期版本号 / v1-v3）**。

#### 7.1 视觉规范审计（v5.1+，麦肯锡风硬性）

- [ ] **0 emoji**（全文档无 🔴💡📌⭐⚠️🎯 等任何 emoji 字符）

## last30days 能力整合

当任务需要“最近30天真实讨论/热度/用户观点/竞品最新动作”时，优先复用 last30days 的思想：**查询质量预检、实体/账号/社区预解析、2-4 个子查询计划、多平台信号聚合、证据聚类、内联引用与原始结果留档。**

如果本机安装了 `last30days` skill，先读它自己的 `SKILL.md` 并按它的 contract 调用，不要只把它当搜索关键词。适配细节见 `references/deep-research-search-protocol.md`。
