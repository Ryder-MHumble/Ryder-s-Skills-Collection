# 行业调研报告

撰写、评估、整合行业调研报告的完整工作流。核心原则：**每条数据有source，每个判断有依据，没有source就不写。**

## 触发条件

- 用户要求撰写行业/市场/技术调研报告
- 用户要求评估他人的调研报告
- 用户要求整合多份调研报告
- 用户要求给报告「找source」或「补充来源」
- 用户提到「招生筛选」「考生画像」「人才画像」「材料筛选」
- 用户上传考生/申请人的材料包（通常是zip）
- 用户说「帮我分析这个学生的材料」「做一个画像」
- 用户需要为组织调研设计访谈引导提纲/消息（见第七节）
- 用户提到「访谈邀请」「访谈提纲」

## 一、报告撰写流程

### 1. 升维确认

动手前先确认：
- **报告目的**：兴趣探索（粗粒度即可）还是商业决策（必须精确到source可达）？两者对数据精确度要求完全不同
- **汇报对象**：给自己看 vs 给领导汇报 vs 给投资人——决定深度和格式
- **已有材料**：用户是否已有部分报告/数据需要整合

### 2. 结构化撰写

标准结构（可按需裁剪）：
1. **目录**（带锚点跳转）——报告超过3章时必须有
2. **核心判断**（3-5条，每条带WHY）——结论先行
3. **行业定义与分类**——行业是什么、按什么维度分、与相邻品类的边界在哪
4. **行业全景**——市场规模+赛道分化
5. **分赛道/分场景详细分析**——每个玩家带source
6. **用户评价汇总**（正面+负面）——不只列好评，负面评价同样重要
7. **用户需求热度排名**——星级评分+正负面反馈并列
8. **竞品横向对比表**——同赛道核心产品一字排开对比
9. **政策/行业动态**——附原文链接
10. **来源汇总表**——URL+可达性+等级

### 2.5 工业硬件/To B产品专项结构

当调研对象是工业硬件/To B产品（如工业外骨骼、工业可穿戴、安全设备等），在标准结构之上**必须增加**以下模块：

**① 概念厘清（必须放在报告最前面）**

用户给的品类词和用户真正的意图可能不是一个东西（如"工业外骨骼"vs"劳工保护外骨骼"）。**动手前先厘清：**
- 用户说的品类词 = 产品分类？还是使用目的？还是场景？
- 产品分类与使用目的之间的关系是什么（子集？交叉？等同？）
- 在报告中**以第一章的形式**写清概念关系，让读者知道边界在哪
- 删掉不符合用户意图的子品类（如用户说"劳工保护"就删消费级/康复级）

**② 报告结构按维度组织，不按信息来源散列**

⚠️ **常见错误：一会儿写雇主视角，一会儿写产品特性，来回跳。** 正确做法：
- **维度一：市场与渠道**（规模、劳动力基数、法规、保险/工会/雇主谁买单）
- **维度二：产品与企业**（每家企业先2-3句概述"做什么的"，再产品/材料/技术/价格/客户/竞争力表格）
- **维度三：竞争格局与切入点**（横向对比表+空白+建议）
- 不要在同一个章节里既写市场数据又写产品参数

**③ 企业介绍格式（强制）**

每家企业按以下顺序：
1. **概述段**：2-3句话，说清"这家公司做什么的+核心定位"（不是公司历史，是它在市场中的位置）
2. **产品/技术/价格表**：产品名称、技术路线、材料、价格、客户案例、核心竞争力——用表格而非大段文字

**④ 关键结论视觉突出**

报告中最重要的判断（如"这是增速最快的赛道"）必须：
- 在标题/小标题中标🔴或红色加粗
- 在表格中用红色标注
- 不让读者去"找到"核心结论，核心结论要跳出来

**⑤ 量化市场空间（独立板块）**

用标杆企业的具体数字做测算，作为独立章节：
- 标杆企业员工数、仓库数、日处理量等具体数字
- 设定渗透率场景（如5%），计算市场空间 = 人数 × 渗透率 × 单价
- 分场景列表（仓储/建筑/制造等）

**⑥ 商业模式拆解（谁买单？）**

| 买单方 | 逻辑 | 占比 | 典型案例 |
|--------|------|------|----------|
| 雇主/企业自身 | 减赔偿/减缺勤/提产 | 通常>70% | — |
| 工会/行业组织 | 劳资谈判中的安全条款 | 5-15% | 欧洲3PL物流 |
| 保险公司 | 减赔分成 | 新兴<5% | EaaS试水 |
| 租赁/金融服务 | 降低采购门槛 | 5-10% | — |
| 政府/公共采购 | 公共健康/安全投资 | 视地区 | 欧洲职业安全项目 |

**② 定价区间分层**（按技术路线/功能分级列出价格带，而非单一均价）

| 产品类型 | 价格区间 | 代表产品 |
|----------|----------|----------|
| 被动/低配 | $___-___ | — |
| 主动/高配 | $___-___ | — |
| 租赁/订阅 | $___/月 | — |

**③ ROI逻辑量化**（雇主视角为什么愿意买单）
- 单次工伤/事故索赔成本 → 设备售价对比 → 回本周期
- 缺勤率减少 → 产能提升 → 年化收益
- 保险费率变化 → 持续收益

**⑦ 法规/合规环境（To B产品的核心驱动力之一）**
- 只写用户指定的国家/地区，不自行扩展（用户说美/英/法/德就只写这4国）
- 各国/地区差异对比
- 法规趋势（收紧/放宽/标准化中）

**⑧ 竞争格局空白与切入建议**
- 不只是列现有玩家，必须标注"没人做但应该有人做"的空白
- 切入建议要具体到：产品形态、目标市场、定价、渠道

案例：`references/exoskeleton-labor-protection-market.md`

### 2.6 严格裁剪报告范围

**用户给了范围边界，不要自行扩大：**
- 用户只问了3个国家 → 只写那3个国家，不要"顺手"加日本/韩国/其他
- 用户只问了劳工保护场景 → 删掉消费级/康复级产品，不在报告中"留个尾巴"
- 用户删掉某节 → 直接删，不要追问"确定删吗"

**判断标准：**如果用户明确说了范围（国家/场景/品类），超出范围的内容=噪音。用户需要时会自己说"再加XX"。

### 3. Source分级标注

每条数据必须标注来源等级：

| 等级 | 定义 | 示例 | 标注方式 |
|------|------|------|---------|
| A级 | 官方文件/监管披露 | SEC EDGAR 10-K、公司IR财报PDF、政府公文 | **Source**: URL |
| B级 | 权威媒体报道 | TechCrunch、Reuters、36氪首发、行业期刊 | **Source**: URL |
| C级 | 公司官网/行业资讯 | 公司产品页、爱范儿、极客公园、机器人大讲堂 | **Source**: URL |

**无source的数据不写进报告。** 宁可标注「未公开」或「暂无可靠来源」，也不编造。

### 3.5 市场份额估算方法（当精确数据不可得）

工业/ToB领域多数企业为私有公司，不披露市场份额。此时采用交叉验证法估算：

**方法**：
1. **公开指标替代**：客户数量、部署量、融资金额、营收估算（上市公司取SEC数据，私有公司取Tracxn/Crunchbase/Similarweb估算）
2. **细分赛道竞品数量**：部分细分仅1-2家玩家（如手部握力增强、可穿戴座椅），按"竞品数量=垄断程度"推算
3. **研究机构分层**：QY Research、Grand View Research等的市场规模分层数据×企业公开指标占比

**标注规则**：
- 估算份额必须标注"估计"字样和口径（按收入/按部署量）
- 必须附方法论说明段落，告知读者数据来源和推算逻辑
- 如果某企业完全无公开数据可交叉验证，标注"数据不可得"而非编造

**案例**：外骨骼劳工保护调研中，8家国际企业均无官方市场份额披露。通过(1)客户数+部署量+融资金额、(2)细分竞品数、(3)机构市场数据三层交叉，得出各细分份额估算。

### 4. 诚实标注数据边界

报告末尾必须有「数据边界说明」，明确：
- 哪些数据是A级验证的
- 哪些是C级（可能偏差）
- 哪些领域因公开信息有限而缺失
- 不在报告中罗列搜索工具受限的借口

### 5. 竞争格局横向对比

当用户要求「Top N公司对比」/「竞品对比」/「国内玩家分析」时，按以下流程：

#### Step 1：锁定候选名单
- 从用户已有报告中提取已提及的公司
- 从用户口头提及的品牌中提取（注意：用户提到的品牌名可能是新品牌/内部代号，搜索引擎零结果 ≠ 不存在）
- 跨赛道考虑：Top N不一定全是纯赛内公司，跨界入局者同样重要

#### Step 1.5：跨界入局者专项排查

当用户要求排查「跨界入局者」（非本赛道主业公司进入目标赛道）时，按行业相邻度搜索：

**相邻行业优先级**（以人体运动类产品为例）：
1. **汽车制造** → 产线工人外骨骼是天然场景（现代X-ble、本田Walking Assist）
2. **消费电子** → 老年康养+健康生态延伸（三星Bot Fit、松下ATOUN）
3. **户外运动品牌** → 产品升级+品类扩展（始祖鸟MO/GO）
4. **家电/康养** → 场景+渠道打包方案（海尔W1/W2 via HSCube）

**排查方法**：
- 对每个行业选Top 3品牌，Yahoo搜索「{品牌名} exoskeleton」
- 验证证据链：官网产品页 > 行业垂直站报道 > 主流媒体报道 > Alamy/Getty图片caption
- **知名品牌零结果≠不存在**：可能是子公司/子品牌运营（如海尔→HSCube），用Yahoo图片搜索验证
- 排查未果的品牌也要列出（避免未来重复搜索），标注❌

**跨界入局模式归类**（已验证案例）：
- **自研自用→外溢**：现代（产线用→卖给别人）
- **投资孵化**：松下→ATOUN
- **联合开发**：始祖鸟×Skip（品牌+技术互补）
- **场景打包**：海尔（外骨骼作为康养方案组件）
- **品牌降维**：三星（医疗设备→消费品定位）

详见 `references/exoskeleton-cross-industry-entrants.md`

#### Step 2：逐公司定向验证（优先级排序）

**中国公司信息获取优先级**（搜索环境受限时的实战策略）：
1. **直接访问公司官网** — 最可靠的C级来源。用browser导航（很多中国公司官网是SPA，curl拿不到内容）
2. **Yahoo搜索** — Google/Bing均不可用时唯一有效的搜索引擎。英文查询+中文品牌名混合效果最好。图片搜索的Alamy/Getty图库caption是产品发布事件的有力证据
3. **YouTube频道页** — 新兴消费品牌的验证来源，频道About含公司介绍+社媒链接矩阵
4. **已知媒体搜索** — 人民网/第一财经/36氪/爱范儿（见source验证策略）
5. **社交平台** — 小红书/抖音/微信公众号全部不可用

⚠️ **公司官网不可达 = 信息盲区**：部分中国公司官网长期无法访问（DNS失败/超时/空响应），在当前搜索环境下几乎无替代信息源。此时只能：
- 标注为「⚠️ 信息未独立验证，基于行业公开信息推断」
- 询问用户是否有该公司的官网链接或最新材料

#### Step 3：构建对比表

标准维度（按需裁剪）：

| 维度 | 公司A | 公司B | ... |
|------|-------|-------|-----|
| 成立时间/地点 | | | |
| 产品线 | | | |
| 核心技术差异 | | | |
| 目标市场 | | | |
| 融资/估值 | | | |
| 出货量/市占 | | | |
| 置信度 | ✅/⚠️/❌ | | |

**置信度标注**：
- ✅ 官网可访问+信息已验证
- ⚠️ 官网不可达/信息未独立验证，基于行业公开信息推断
- ❌ 无任何可验证信息源

#### Step 4：标注缺口，回问用户
- 列出未能验证的品牌/数据点
- 明确请用户提供：品牌中文全名、官网链接、电商平台链接
- 不要在无目标的web搜索上浪费超过10分钟

## 二、评估外部报告并整合

当用户提供他人的调研报告，按以下流程处理：

### Step 1：通读全篇，分类标记

对每个信息点标记：
- ✅ **好**：有具体数据+可验证来源 → 保留，去验证source
- ⚠️ **存疑**：有数据但无来源/来源不可达 → 尝试找source，找不到则删
- ❌ **坏**：无数据纯推测/来源验证失败/自相矛盾 → 删除

### Step 1.5：识别外部报告的优点（不是只挑错）

评估外部报告时，**先识别它做得比我好的地方**，再判断是否吸纳：
- 是否有我的报告缺少的维度？（如负面评价、用户需求星级评分、竞品对比表）
- 是否有更细粒度的分析？（如场景需求分布、易混淆公司澄清）
- 是否有我遗漏的数据点或source？

**吸纳原则**：好的吸纳，坏的不吸纳。具体：
- 外部报告有而我没有的**有价值维度** → 吸纳进新版
- 外部报告有而我没有的**已验证source** → 直接引用
- 外部报告信息冗余或低质量的部分 → 不吸纳，不花篇幅

**易混淆公司澄清**：如果两/多家公司常被混淆（如程天科技 vs 傲鲨智能），在报告中做明确区分说明，不要假设读者知道它们不同。

### Step 2：验证source可达性

对报告中引用的URL逐一验证：
```bash
curl -sI -L "$URL" -H 'User-Agent: Mozilla/5.0' --max-time 12 -o /dev/null -w '%{http_code}'
```
- 200/301/302 = ✅ 可达
- 404/403/000 = ❌ 不可达

### Step 3：为「好的」数据补充source

如果原文数据有价值但缺source：
1. 先用curl验证文中已有URL
2. 用browser搜索已知网站（如36氪搜公司名+融资）
3. 在搜索结果页用browser_snapshot提取文章URL
4. 导航到具体文章验证数据（⚠️36氪文章页有验证码，搜索页标题确认即可）
5. 找到source后标注等级

### Step 4：整合进自己的报告

- 好数据（已验证source）：整合进对应章节，标注source等级
- 好数据（找不到source）：在报告中标注为「C级/未独立验证」
- 坏数据：删除，在报告末尾「对原报告评判」中说明删了什么及原因
- 整合后重新生成PDF

### Step 5：输出评判摘要

在报告末尾附上对原报告的评判：
- **做得好的**：列出具体条目
- **做得不好的**：列出具体条目及处理方式（删除/修正）

## 三、Source验证工作流

web搜索在当前环境严重受限（Google/Bing/DuckDuckGo均不稳定），替代策略：

### 策略1：直接URL验证（最可靠）
已知具体URL时，直接curl验证可达性。

### 策略2：已知网站内搜索
- **人民网搜索**：`https://search.people.cn/s/?keyword={keyword}&st=1` — 中国机构/政策研究首选，无验证码，聚合上观新闻/解放日报等权威来源
- **第一财经**：`https://www.yicai.com/search?keys={keyword}` — 科技/产业信息，搜索页可提取文章URL
- **36氪**：`https://www.36kr.com/search/articles/{keyword}` → browser导航 → 提取文章URL ⚠️文章页有验证码
- **SEC EDGAR**：`https://data.sec.gov/api/xbrl/companyfacts/CIK{XXXXXXXX}.json`（CIK+10位补零）
- **爱范儿/极客公园/腾讯新闻**：已知ID格式可直接构造URL，无验证码，是优质B级来源
- **TechCrunch**：直接构造URL或browser导航
- **GlobeNewsWire**：直接构造URL

### 策略3：Browser搜索
用browser_navigate访问已知网站的搜索页，比搜索引擎更有效。

**内容提取技巧**：当 `browser_snapshot` 截断长页面内容时，用 `browser_console` 执行 JavaScript 提取全文：
```
browser_console: document.querySelector('article').innerText
// 或全文：document.body.innerText
// 截取后半部分：document.querySelector('article').innerText.substring(8000)
```
这比反复滚动+snapshot更可靠，尤其适用于文档页、GitHub README、新闻文章等有语义HTML结构的页面。

### 不做的事
- 不用Bing搜中文（返回结果严重偏移，如搜「傅利叶融资」返回游戏）
- 不信任delegate_task子agent返回的URL（可能编造）
- 不在报告中写搜索工具受限的借口
- 不给delegate_task子agent分配browser工具集的web搜索任务（大概率超时，600秒限制不够）

### 多维度调研的委派策略
当调研报告涉及5+个并行维度时：
- **terminal密集型任务**（数据计算、文件操作、API查询）→ 委派成功率高，优先委派
- **web搜索密集型任务**（需要大量web_search/web_scrape）→ 委派超时率约50%，建议缩小每个子agent的搜索范围，或直接在主agent中基于领域知识补充
- **fallback模式**：先并行委派所有维度→等待完成的维度→对超时的维度，评估是重新委派（缩小范围）还是直接补充（基于领域知识+少量验证）→通常"直接补充+诚实标注数据边界"比反复重试更高效

### 5. 交付前source URL批量验证（硬性步骤）

**用户零容忍不可达source。** 在生成PDF之前，必须对所有将写入报告的source URL做批量可达性验证，只保留HTTP 200/301/302的URL。

```bash
# 批量验证脚本（写入报告前必跑）
urls=(
  "https://www.hyundai.com/worldwide/en/..."
  "https://fda.report/PMN/K181294"
  "https://hscube.com/"
)
for url in "${urls[@]}"; do
    code=$(curl -sL -o /dev/null -w '%{http_code}' --max-time 10 "$url" 2>/dev/null)
    if [ "$code" = "200" ] || [ "$code" = "301" ] || [ "$code" = "302" ]; then icon="✅"; else icon="❌"; fi
    echo "$icon $code $url"
done
```

**规则**：
- ❌ 404/403/000 的URL **不写进报告**，即使研究阶段该URL曾经可达
- 研究阶段找到但验证时404的URL（如Arc'teryx博客原文已404），改用二级source（如行业垂直站报道引用该事件）或标注「原文已下线，事件由XX报道确认」
- 403但内容已通过browser确认的URL可保留，但标注「需浏览器访问」

## 六、常见陷阱

- **研究阶段URL交付时可能404**：产品发布博客/新闻稿经常在数月后变更URL或下线（如Arc'teryx MO/GO官方博客2024.7发布→2026.5访问时已404）。**交付前必须重新验证每个URL**，不能假设研究阶段验证过的URL仍然可达
- **编造URL**：找不到source时宁可标注「未公开」，不要编造看起来合理的URL
- **百度百科404**：百度百科链接编码方式不稳定，经常404，不能当可靠source
- **36氪文章URL不可猜测**：必须通过搜索页找到真实文章ID，格式为 `/p/{数字ID}`
- **SEC EDGAR需User-Agent**：curl必须加 `-H 'User-Agent: ...'`，否则被拒绝
- **SEC EDGAR XBRL zip URL返回403**：`efts.sec.gov/LATEST/ots_xbrl_instances/{id}-xbrl.zip` 从服务器直接访问可能返回403。改用EDGAR搜索页链接：`https://www.sec.gov/cgi-bin/browse-edgar?action=getcompany&CIK={CIK}&type=10-K`，搜索API确认文件存在即可
- **CIK补零**：SEC API格式为CIK+10位补零，如 `CIK0001549084`
- **新兴消费品类搜索盲区**：当目标品类是1-2年内才出现的新兴消费品类（如2024-2025年消费级外骨骼），信息主要存在于小红书/抖音/微信公众号等社交平台——这些平台在此环境全部不可用（验证码/反爬/封IP）。主流搜索引擎（Bing/Google/百度）和科技媒体（36氪/爱范儿）可能尚未收录。**正确做法：先向用户确认品牌中文名、官网链接、电商平台链接，再做定向验证**。不要在无目标的web搜索上浪费超过10分钟。用户提到的品牌名可能是内部代号或新品牌，搜索引擎零结果不等于品牌不存在——此时必须回问用户。
- **知名品牌跨品类提及陷阱**：用户可能提到知名品牌（如海尔）「已经在推广」某品类产品，但搜索引擎和官网均未找到该品牌的相关产品线。可能原因：①品牌仅做概念展示/展会demo未量产；②品牌投资/孵化了子品牌而非自己出产品；③用户信息源是社交平台（此环境不可用）。**正确做法：不要假设知名品牌一定有该品类产品，用Yahoo搜索+图片搜索验证（Alamy/Getty图库caption常含产品发布事件照片），再访问可能运营该产品线的子公司/子品牌官网**。案例：海尔外骨骼→通过Yahoo图片搜索发现Alamy caption确认2025年上海发布→发现运营主体为HSCube（hscube.com），而非海尔集团官网本身。
- **过度展开无信息条目**：如果某公司/品牌在目标市场无实质信息，一句话说明即可，不要花篇幅区分混淆（例：程天科技/傲鲨智能/Dnsys X1在消费级外骨骼市场信息极度匮乏，原报告花了几十行全是「信息有限」「无法获取」）
- **大段搬运证言/评价**：用户Demo证言、媒体引用的原话不要逐条列举，应提炼为正负面比例+代表性结论。7条英文原文逐行搬运是搬运不是分析
- **调研局限性说明**：不要在正式报告中罗列搜索工具受限的借口，那是内部笔记不是交付物
- **中国工商信息查询**：天眼查/企查查/企查查/国家企业信用信息公示系统在此环境全部被封IP，无法查询股东/法人/持股等工商数据。如需此类信息，必须请用户自行查询后提供
- **海外机构调研**：OpenAlex API是发现海外学术论文的最可靠渠道（免费无认证，终端curl可用），搜`api.openalex.org/works?search={关键词}&per_page=25`。学术论文PDF是海外机构制度信息的主要来源——官网和政府法规网站常不可达
- **PDF下载验证**：下载PDF后必须验证有效性——①`file`命令确认文件类型非HTML；②`pdftotext`或`pymupdf`确认页数>0。常见陷阱：URL看起来是PDF但重定向到出版社付费墙/登录页（HTML），pymupdf打开某些合法PDF显示0页（改用pdftotext）
- **宽面市场调研搜索策略**：当需要对一个市场做全景扫描（10+家公司）时，不要逐家串行搜索。正确做法：**第一轮4-6个并行搜索覆盖不同维度**（如：市场整体数据+品类头部玩家+区域市场+商业模式），从第一轮结果中提取候选名单；**第二轮对重点公司做4-6个并行定向搜索**（公司名+产品、公司名+客户案例、公司名+定价/商业模式等）；避免超过2轮，后续靠领域知识补充+诚实标注边界

## 七、组织调研访谈准备（Stakeholder Interview Outreach）

当任务是为组织内部调研设计访谈引导提纲/消息时，使用以下工作流。核心场景：对业务理解有限，需要通过访谈收集各领域负责人的真实关注点。

### Step 1：策略选择——「预选项」还是「引导思考」

- **预选项策略**：你对该领域业务理解足够深 → 提供0.5草案让对方做选择题
- **引导思考策略**：你对该领域业务理解不够，预选项可能偏 → 抛锚点让对方自己长出关注点
- ⚠️ **判断标准**：如果你对各中心业务是"完全不理解"的，预选项不可行——硬给选项会把对方框在错误方向

### Step 2：提纲结构模板

每条消息四段式：

1. **背景段（统一）**：2-3句，说清要做什么+为什么找你。不用项目内部名称（如"风险看板"），用中性功能描述（"数据看板"）
2. **业务锚点（差异化）**：1-2句，说"我们粗浅的了解是……如果方向大致对的话——"。语气=锚点不是选项，给对方纠正空间
3. **思考引导（差异化）**：分三层——①日常最频繁过问什么 ②什么信号需要关注 ③什么标准决定需要院领导层面关注。每层1-2题，总5-6题
4. **收尾（统一）**：一句谦逊收束+"近期当面请教"

### Step 3：层级语气校准

| 读者层级 | 语气 | 开头 | 收尾 |
|---------|------|------|------|
| 高于己方领导 | 谦逊请教 | "打扰了"+"想先请教您" | "当面请教，感谢您的时间" |
| 同级/下级 | 专业直接 | 直接说事 | "期待交流" |

**禁止**：对上级用"我们需要你的判断"这类暗示对等位置的措辞。

### Step 4：敏感词分区

不替受访者定义"什么算风险"——这正是要访谈出来的。统一用中性措辞，让受访者用自己的语言定义：

| 部门类型 | 替代措辞 | 避免 |
|---------|---------|------|
| 政治敏感（党建/纪检） | "需要关注的事项""信号" | "风险""预警""出事" |
| 培养相关（教科人） | "需要关注的事项" | "风险" |
| 内容交付型（战略/秘书） | "信号""关注" | "风险" |
| 有风险管理语义的（行政/科研/产业） | "风险"可自然出现 | — |

### Step 5：多中心负责人消息合并

同一人分管多个中心时，合并为一条消息：共用开头+收尾，各中心问题用小标题分隔。不要给同一领导发多条重复背景的消息。

### Step 6：验证清单

- [ ] 每个中心的业务锚点是否列出了下属部门名称？
- [ ] 姓名是否核实（常见姓氏错误：邵≠绍）？
- [ ] 多中心负责人是否合并为一条消息？
- [ ] 敏感词是否按分区处理？
- [ ] 语气是否匹配层级？

详细案例和敏感词对照表见 `references/stakeholder-interview-outreach.md`
完整消息实例见 `references/two-academy-dashboard-example.md`（两院数据看板项目6条实际消息，含行政管理中心完整消息+多部门合并示例+设计决策记录）

### Step 7：思考引导设计原则

1. **锚点不是选项**——业务锚点的作用是降低认知启动成本，不是框定方向
2. **每个问题带方向提示但不给选项**——"进度偏差、经费问题、人员变动，还是其他？"（给方向启发+留"其他"出口）
3. **分层递进**——日常（轻松启动）→ 信号（核心挖掘）→ 标准（收束判断）
4. **第3题是最关键的**——"什么算需要领导关注的信号"正是整个访谈要挖掘的核心

### Step 8：常见陷阱

- 给了业务锚点但语气太自信 → 对方觉得你已下定论，不需要补充 → 用"粗浅""似乎""大致"软化
- 问题太开放 → 对方不知从何想 → 每题给方向提示
- 问题太具体 → 变相回到预选项 → 每题末尾加"还是其他？"
- 发给同一领导两条消息 → 对方困惑 → 先查是否有跨部门负责人，合并
- 敏感词按自己部门习惯用 → 对方不适 → 按对方部门属性选词
- 称呼打错名字 → 尴尬 → 双重核对，特别是生僻姓氏（邵≠绍）

## 八、招生人才画像（Talent Profiling for Admissions）

当场景为招生筛选时，使用以下专项工作流。核心价值：将170页考生材料压缩成1页结构化档案卡，让研究员10秒出判断。

### Step 0：确认现状——已有系统时，先看反馈再设计新功能

**⚠️ 跳过反馈直接做画像/可视化，是本项目已犯过的最大错误。**

当用户提到「优化」「迭代」「反馈」等词，或隐含已有筛选系统时：
1. **找反馈数据**：问用户要反馈意见表/汇总/群聊记录
2. **结构化分类反馈**（按紧迫度排序）：
   - **A. 交互流程问题**（最高频=最痛=最优先改）——工作流断裂、操作步骤断、状态不够用
   - **B. 信息抽取准确性**（直接影响信任度）——幻觉、误判、漏抽
   - **C. 筛选逻辑与权重**——阈值不对、硬性筛选缺失
   - **D. 展示与布局**——信息密度、对比能力
   - **E. 性能与集成**——响应慢、数据源没打通
3. **映射到需求模块**：P0=筛选模块（修好现有流程），P1=画像模块，P2=报告模块
4. **核心教训：7人29条反馈中，6人抱怨筛选流程断裂，0人抱怨画像不够好——因为还没用到画像。P0永远是修好现有流程，不是做新功能。**

详见 `references/academic-talent-feedback-analysis.md`

### Step 1：材料包解压与清点

```bash
mkdir -p /tmp/student_material && cd /tmp/student_material
unzip -o "/path/to/考生材料包.zip"
ls -la /tmp/student_material/*/
```

输出材料清单表（文件、页数、类型、数据可提取性）。**关键发现：材料包中往往1-2个文件占90%页数，但真正有判断价值的结构化信息不到1页。**

### Step 2：多格式文档信息提取

- **文字PDF**：`pymupdf`直接提取（⚠️ 1.25+版import名称为`pymupdf`而非`fitz`）
- **扫描件PDF**：pymupdf渲染（2x放大）+ pytesseract OCR（需`apt install tesseract-ocr tesseract-ocr-chi-sim`）
- **大体积科研材料**：只提取目录页和标题页的结构化信息，不要全文提取

### Step 3：构建结构化画像数据

从所有材料中提取并交叉验证：基础信息、学术产出（论文/会议/项目）、研究方向、推荐信评价、荣誉、英语、风险标注。

**交叉验证重点**：简历GPA vs 成绩单实际分数、简历论文列表 vs 科研材料中实际收录、推荐信评分 vs 其他材料展现的水准。

### Step 4：五维评分画像卡

| 维度 | 权重 | 评分依据 |
|------|------|----------|
| 学术能力 | 30% | 论文数量/作者排序/发表/会议级别 |
| 研究潜力 | 25% | 研究计划质量+方向匹配度+自驱学习证据 |
| 基础素养 | 20% | GPA/排名/核心课成绩/英语/竞赛 |
| 材料完整度 | 15% | 材料是否齐全+信息是否自洽 |
| 风险指数 | 10% | 材料矛盾/信息缺失/异常信号（反向分） |

⚠️ **评分标准必须用真实正例+反例校准**，不要凭空定义阈值。

### Step 5：可视化原型（HTML+SVG，投屏级质量）

三类图表，用纯HTML+CSS+SVG构建，无需前端框架：
- **档案卡**：深色banner+指标行+信息网格+风险标注+综合评级
- **雷达图**：SVG五轴雷达，支持多人overlay对比
- **热力图**：HTML表格+色阶背景，行=学术产出等级/列=院校层次

截图交付：Playwright + PIL（`pip install playwright Pillow --break-system-packages` + `python3 -m playwright install chromium`）

模板见 `references/academic-talent-profile-visualization-prototype.html`

### Step 6：对齐方案

核心原则：**产品经理不是需求提问者，是带着方案来确认的人。** 对齐方案必须包含：报告结构草案、具体图表原型、DDL倒推的前置依赖、需要业务方提供的校准数据。

**需求文档≠交付范围**：业务方给的需求文档通常是全链路愿景，但DDL是硬的。必须明确区分「6月做的」和「长期做的」。
